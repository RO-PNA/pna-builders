# v0.3 — Knowledge Graph (Categories, Tags & Ontology)

> **Version**: v0.3 / **Status**: Draft / **Depends on**: v0.2.1 (Auth hardening, admin role) / **Next**: v0.4 (Auto relation extraction)

Knowledge를 "최신순 스트림"에서 **"카테고리·태그로 필터 가능하고, 개념·인물이 엮인 지식 그래프"**로 전환한다. 두 축(카테고리·태그 / 온톨로지)은 데이터 모델이 직접 맞물리므로 하나의 릴리즈로 묶는다.

---

## 1. 배경 & 문제 정의

### 현재 상태 (v0.2)
- `items.category`는 **단일 자유 문자열** (디폴트 "Article"). 스키마에 enum 없음
- 필터/검색 UI 없음, 홈은 최신순 리스트만
- 태그 개념 부재
- 콘텐츠·개념·인물 간 관계를 저장할 방법 없음
- 같은 개념이 여러 표기로 쪼개짐 (예: "LLM" vs "대규모 언어모델")

### 문제
- 아이템이 쌓일수록 **발견성 저하** — 주제별 탐색 불가
- GeekNews 대비 차별점인 **복수 자료 연결**을 살리려면 의미 태그·그래프가 필수
- 학습자가 주제를 **맥락적으로** 탐색할 수 없음

### 목표
1. 정규화된 카테고리/태그 체계 확립 → 필터·랜딩·SEO 기반 확보
2. 카테고리·태그·개념·인물을 **통일된 그래프 노드**로 재사용 → 온톨로지 골격 구축

## 2. 범위 (Scope)

### In-Scope
**Part A — Categories & Tags**
1. 위계형(2단계) 카테고리 (아이템당 1개)
2. 평면형 태그 (아이템당 다중, 최대 5개)
3. 등록·수정 UI, 홈 필터, `/category/[slug]`·`/tag/[slug]` 랜딩

**Part B — Ontology**
1. 통일 노드·엣지 스키마 (`item`/`tag`/`category`/`concept`/`person`)
2. 관계 타입 세트 (tagged_with, belongs_to, is_subtype_of, related_to, precedes, mentions, authored_by, covers)
3. 관리자용 노드/엣지 편집 UI (Form 기반 최소 기능)
4. `/ontology` 탐색 페이지 (react-flow 기반 2D 그래프)
5. 아이템 상세에 "연결된 개념" 섹션 (1-hop)

### Out-of-Scope (v0.4+)
- 전문 검색(Full-text), 퍼지 매칭
- LLM 기반 자동 태그·관계 추천
- 유저 투표/승인 기반 관계 큐레이션
- 3D 그래프, 시간축 시각화, 외부 KG 연동 (Wikidata 등)
- 사용자 개인화 피드

## 3. Part A — Categories & Tags

### 3-1. 카테고리 체계 (제안)
아이템당 **1개 카테고리 필수**. 2단계 위계.

| 대분류 | 소분류 예시 |
|--------|------------|
| AI | LLM, Agents, RAG, 논문, Tool |
| Product | Growth, PMF, Pricing, UX Research |
| Engineering | Frontend, Backend, Data, DevOps |
| Business | 스타트업, M&A, 투자, 실리콘밸리 |
| Community | PNA 내부, 이벤트, 인터뷰 |

- 소분류는 관리자 관리(초기 시드)
- 대분류만 선택해도 등록 가능, 미분류는 "기타"로 자동 편입

### 3-2. 태그 체계 (제안)
- 평면형, 아이템당 최대 5개
- 관리자 승인 불필요 (소셜 태깅)
- 아이템 등록자만 자기 아이템 태그 편집 가능
- 중복/오타 방지: 등록 시 **자동완성**으로 기존 태그 우선 노출
- 표기 규칙: 소문자, 공백은 하이픈(`-`), 한글 허용 (예: `llm-agents`, `pna-워크샵`)

### 3-3. UX 변경점

**`/submit` (등록·수정)**
- 대분류 → 소분류 2단계 드롭다운
- 태그 자동완성 입력 필드 (Enter로 추가, 신규 태그는 "새로 생성" 안내)

**`/` (홈)**
```
┌────────────────────────────────────────┐
│ Knowledge                    [submit]  │
├────────────────────────────────────────┤
│ [전체] [AI] [Product] [Eng] [Biz]      │  ← 카테고리 칩
│ 🏷 #llm-agents #rag #pmf ...          │  ← 인기 태그 (상위 N)
│ (필터 적용 결과) ...                    │
└────────────────────────────────────────┘
```
- URL 쿼리스트링으로 필터 상태 보존 (`?category=ai&tag=rag`)

**`/item/[id]` (상세)**
- 타이틀 아래 카테고리 배지 + 태그 칩
- 각 칩 클릭 시 해당 필터 페이지로 이동

**새 페이지**
- `/category/[slug]` — 해당 카테고리 아이템 + 소분류 서브필터
- `/tag/[slug]` — 해당 태그 아이템 + 관련 태그 추천

## 4. Part B — Ontology

### 4-1. 노드 타입

| 타입 | 출처 | 예시 |
|------|------|------|
| `item` | `items` 테이블 | 아이템 #123 |
| `tag` | `tags` 테이블 (Part A) | `llm-agents` |
| `category` | `categories` 테이블 (Part A) | AI > Agents |
| `concept` | 신규 | "Tool Use", "Multi-step Reasoning" |
| `person` | 신규 | "Andrej Karpathy", PNA 멤버 |

### 4-2. 관계 타입

| 타입 | 방향 | 의미 |
|------|------|------|
| `tagged_with` | item → tag | 아이템이 태그를 달았다 |
| `belongs_to` | item → category | 아이템의 카테고리 |
| `is_subtype_of` | concept → concept | 하위 개념 |
| `related_to` | concept ↔ concept | 유사/연관 (무방향) |
| `precedes` | concept → concept | 선후행 (학습 순서) |
| `mentions` | item → concept / person | 아이템이 언급 |
| `authored_by` | item → person | 작성자 |
| `covers` | tag → concept | 태그가 특정 개념을 커버 |

### 4-3. UX — `/ontology`
```
┌─────────────────────────────────────────┐
│ Ontology                                │
│ [노드 타입 필터] [관계 필터] [검색]     │
│                                         │
│  ┌───────────────────────────────────┐  │
│  │      (force-directed graph)       │  │
│  │       ●──────● ──────●            │  │
│  │      concept  tag                 │  │
│  │     ●────── ●                     │  │
│  │    item    person                 │  │
│  └───────────────────────────────────┘  │
│  노드 클릭 → 우측 사이드바에 상세        │
└─────────────────────────────────────────┘
```
- 초기 뷰: 연결도 기준 상위 N개 노드 (과밀 방지)
- 1-hop 확장, 드래그·줌·팬 지원
- 라이브러리: **`react-flow`** 우선 검토 (cytoscape.js 대비 번들 크기·React 친화도 우위)

### 4-4. UX — `/admin/ontology` (관리자 전용)
- concept/person CRUD (item/tag/category는 원본에서 관리)
- 엣지 추가/삭제 (from/to/relation 선택)
- 대량 가져오기(CSV)는 v0.4+

**선행 필수**: v0.2.1에서 관리자 권한 플래그 도입(`profiles.is_admin` 또는 별도 테이블). 자세한 내용은 `v0.2.1_google_oauth_debugging.md` 참고.

### 4-5. UX — 아이템 상세 "연결된 개념" 섹션
- 해당 아이템이 연결된 1-hop 노드를 카드 그리드로 표시
- 카드 클릭 → 해당 노드의 서브그래프 뷰

## 5. 데이터 모델

### 5-1. Part A — 새 테이블

```sql
-- 카테고리 (위계형)
create table public.categories (
  id bigint generated by default as identity primary key,
  slug text unique not null,
  name text not null,
  parent_id bigint references public.categories(id),
  sort_order int default 0,
  created_at timestamptz default now()
);

-- 태그 (평면형)
create table public.tags (
  id bigint generated by default as identity primary key,
  slug text unique not null,
  name text not null,
  usage_count int default 0,
  created_at timestamptz default now()
);

-- 아이템 ↔ 태그
create table public.item_tags (
  item_id bigint references public.items(id) on delete cascade,
  tag_id bigint references public.tags(id) on delete cascade,
  created_at timestamptz default now(),
  primary key (item_id, tag_id)
);

-- items에 category_id 추가
alter table public.items
  add column category_id bigint references public.categories(id);
```

### 5-2. Part B — 새 테이블

```sql
-- 통합 노드 (item/tag/category는 원본 테이블 참조)
create table public.ontology_nodes (
  id bigint generated by default as identity primary key,
  node_type text not null check (node_type in ('item','tag','category','concept','person')),
  ref_id bigint,          -- item/tag/category일 때 원본 PK
  slug text unique,       -- concept/person 고유 slug
  name text not null,
  description text,
  created_at timestamptz default now()
);
create index on public.ontology_nodes (node_type, ref_id);

-- 엣지
create table public.ontology_edges (
  id bigint generated by default as identity primary key,
  from_node bigint references public.ontology_nodes(id) on delete cascade not null,
  to_node bigint references public.ontology_nodes(id) on delete cascade not null,
  relation text not null check (relation in (
    'tagged_with','belongs_to','is_subtype_of','related_to',
    'precedes','mentions','authored_by','covers'
  )),
  weight numeric default 1.0,
  created_at timestamptz default now(),
  unique (from_node, to_node, relation)
);
create index on public.ontology_edges (from_node, relation);
create index on public.ontology_edges (to_node, relation);
```

### 5-3. 동기화 규칙
- 아이템/태그/카테고리 생성 시 → `ontology_nodes` 자동 insert (API 레이어 또는 트리거)
- `item_tags` insert 시 → `tagged_with` 엣지 자동 생성
- `items.category_id` 업데이트 시 → `belongs_to` 엣지 자동 생성
- concept/person 노드는 `/admin/ontology`에서 수동 등록

### 5-4. 파생 뷰 (옵션)
```sql
create view public.item_graph as
select
  i.id as item_id,
  n.id as node_id,
  n.node_type,
  n.name,
  e.relation
from public.items i
join public.ontology_nodes n_item on n_item.node_type = 'item' and n_item.ref_id = i.id
join public.ontology_edges e on e.from_node = n_item.id
join public.ontology_nodes n on n.id = e.to_node;
```

### 5-5. RLS
- `categories`, `tags`, `ontology_nodes`, `ontology_edges` — 공개 읽기
- `categories`, `ontology_nodes`, `ontology_edges` insert/update/delete — **관리자 전용**
- `tags` insert — 인증 유저 허용 (소셜 태깅)
- `item_tags` — 아이템 소유자 또는 관리자

## 6. 마이그레이션 전략

1. `categories` 시드 입력 (대분류 5 + 초기 소분류)
2. 기존 `items.category` 텍스트를 slug 매칭으로 `category_id`로 변환, 매칭 실패분은 "기타"로
3. 아이템 ↔ 노드 자동 생성 배치 (`items` 전체 순회)
4. 안정화 확인 후 `items.category` 텍스트 컬럼 제거
5. `concept`, `person` 초기 시드 등록 (§10 참고)

## 7. API 변경

| 엔드포인트 | 기능 |
|------------|------|
| `GET /` (서버 컴포넌트) | `?category`, `?tag` 쿼리 지원, join 쿼리 필터링 |
| `POST /api/items` | body에 `category_id`, `tags: string[]` 수용. 태그 upsert 후 `item_tags` 연결 + 온톨로지 노드·엣지 동기화 |
| `GET /api/tags` (신규) | 자동완성용 prefix 검색 |
| `GET /api/categories` (신규) | 카테고리 트리 반환 (5분 캐시) |
| `GET /api/ontology/graph` (신규) | 필터·검색에 맞는 서브그래프 `{ nodes, edges }` |
| `GET /api/ontology/node/[id]` (신규) | 노드 상세 + 1-hop 이웃 |
| `POST /api/ontology/nodes` (신규) | concept/person 생성 (관리자) |
| `POST /api/ontology/edges` (신규) | 엣지 생성 (관리자) |

## 8. 단계별 릴리즈

| 단계 | 내용 | 기준 |
|------|------|------|
| Step 1 | **Part A 스키마·마이그레이션** | 기존 아이템 100% 매핑 |
| Step 2 | **Part A 등록/수정 + API** | `/submit`에서 카테고리·태그 저장 |
| Step 3 | **Part A 홈 필터 + 랜딩** | `/`, `/category/[slug]`, `/tag/[slug]` 동작 |
| Step 4 | **Part B 스키마 + 동기화** | `items`·`tags`·`categories` 노드 자동 생성 |
| Step 5 | **Part B 관리자 UI** (`/admin/ontology`) | concept/person 수동 추가 가능 |
| Step 6 | **Part B 탐색 페이지** (`/ontology`) | react-flow 그래프 뷰 |
| Step 7 | **Part B 아이템 상세 "연결된 개념"** | 1-hop 이웃 카드 렌더 |

## 9. 성공 지표

**Part A**
- 아이템 등록 시 카테고리 설정 비율 **100%** (필수화)
- 아이템 등록 시 태그 ≥ 1개 부착 비율 **≥ 70%**
- 주간 홈 방문자의 필터 사용률 **≥ 30%**
- `/category`, `/tag` 랜딩 페이지 1주 내 검색엔진 인덱스

**Part B**
- v0.3 완료 시 `concept` 노드 **≥ 50개**, `person` 노드 **≥ 20개**
- 아이템 평균 연결 엣지 수 **≥ 3**
- `/ontology` 주간 방문자 중 노드 클릭 **≥ 5회/세션**
- 아이템 상세 "연결된 개념" 카드 CTR **≥ 10%**
