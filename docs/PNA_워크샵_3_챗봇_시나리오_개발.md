## 핵심 요약 (Executive Summary)

### 이 문서가 다루는 것

**Product Autopsy**는 "성장이 멈춘 제품을 부검하여 근본 원인을 밝혀내는" 팀 기반 문제 정의/구조화 시뮬레이션 게임입니다. AI 챗봇이 게임마스터 역할을 하며, 팀은 6개 Phase를 거쳐 가상 서비스의 성장 정체 원인을 진단하고 개선 전략을 수립합니다.

### 게임 한눈에 보기

| 항목 | 내용 |
|------|------|
| 소요 시간 | 약 120분 |
| 팀 구성 | 3~5인/팀, 최대 4팀 |
| 학습 범위 | 문제 정의/구조화 (실행 단계 제외) |
| 도메인 | A.핀테크 간편결제 / B.이커머스 신선식품 멤버십 / C.콘텐츠 숏폼 / D.애드테크 광고 SaaS |
| 팀 운영 | 오프라인 토론 → 합의 → 서기(로테이션)가 챗봇에 제출 |
| 최종 산출물 | 1-Pager PRD + PPT 생성용 프롬프트 + 결과 시뮬레이션 자동 출력 |
| 피칭 | P6 종합 후 오프라인으로 진행 (게임 외부) |

### Phase 구조 × 프레임워크 × 산출물

| Phase | 핵심 질문 | 프레임워크 | 시간 | 산출물 |
|-------|----------|-----------|------|--------|
| P0 | — | — | 5분 | 팀 등록 |
| **P1 문제 감지** | 무슨 일이 벌어지고 있는가? | **S-C-Q-A** | 15분 | S-C-Q 카드 |
| **P2 문제 탐색** | 구체적으로 무엇이 문제인가? | **질문·데이터 요청·VIP 인터뷰** | 15분 | 핵심 문제 정의서 |
| **P3 문제 분해** | 이 문제는 어떤 구조인가? | **MECE + Logic Tree** | 15분 | 문제 트리 |
| **P4 원인 추적** | 왜 이 문제가 발생하는가? | **5-Whys + AARRR/RARRA/Growth Loop** | 20분 | 성장 진단서 |
| **P5 가설 검증** | 어떻게 해결할 수 있는가? | **If-Then-Because** | 25분 | 검증된 가설 카드 |
| **P6 종합** | 어떤 성장 전략을 실행할 것인가? | — | 15분 | 성장 전략서 + 결과 시뮬레이션 |

### VIP 만족도 시스템

도메인별 3인의 이해관계자(VIP)가 존재. 팀 의사결정마다 만족도가 변동하고, Phase 진행 조건 + 엔딩 분기에 영향. 모든 VIP를 동시에 만족시키는 정답은 없음. VIP는 Phase별로 한 명씩 활성화되며, 동시에 2명 이상 등장하지 않음.

### 무료→유료 연결

무료 워크샵에서 체험한 프레임워크(S-C-Q-A → 질문·탐색 → MECE → 5-Whys + 성장 프레임워크 → If-Then-Because)를 유료 워크샵에서 AI 에이전트 시스템 프롬프트에 내장 → "나만의 AI 기획 에이전트"를 만드는 유료 워크샵으로 전환.

## 0. 설계 원칙

### 0-1. 교육적 설계 근거

| 방법론 | 출처 | 게임에서의 역할 |
|--------|------|----------------|
| **McKinsey 7-Step** | Conn & McLean, *Bulletproof Problem Solving* (2018) | Phase 순서 (정의→탐색→분해→원인→가설→종합) |
| **Kolb Experiential Learning** | Kolb, *Experiential Learning* (1984) | 매 Phase 안에서 체험→성찰→개념화→실험 미니 루프 |
| **Jonassen Ill-Structured Problem** | Jonassen (1997, 2000) | VIP 3인의 상충 관점으로 정답 없는 의사결정 강제 |
| **TDCC DEI Pattern** | 토스 메이커스 (2025) | P1-3 = Description, P4-5 = Execution, P6 = Improvement |

### 0-2. 챗봇 정체성: 게임마스터(GM)

| 챗봇이 하는 것 | 챗봇이 하지 않는 것 |
|----------------|---------------------|
| 도메인 상황을 시뮬레이션한다 | 정답을 알려주지 않는다 |
| 팀 선택에 따라 VIP 반응과 결과를 다르게 보여준다 | 프레임워크를 사전에 지정하지 않는다 |
| 팀 행동을 사후에 프레임워크로 해석해준다 | 강의하지 않는다 |
| 예상과 다른 데이터를 섞어 깨달음을 유도한다 | 모든 VIP를 동시에 만족시키는 선택지를 주지 않는다 |

### 0-3. 팀 운영 방식

```
[팀 4명] → 개인 작성(30초~2분) → 돌아가며 발표 → 토론·합의 → 서기가 챗봇에 제출
```

| 규칙 | 내용 |
|------|------|
| **개인 작성 선행** | 모든 Phase에서 토론 전 반드시 개인 작성 시간 확보 (목소리 큰 1명 지배 방지) |
| **서기 로테이션** | 매 Phase마다 챗봇 입력 담당자가 바뀜 → 전원 참여 보장 |
| **구조화 양식** | 자유 텍스트가 아닌 Phase별 정해진 양식에 합의 결과를 채움 |
| **개인 LLM 활용** | Phase 3, 5에서 개인 LLM으로 다양한 관점 탐색 후 팀 토론에 반영 |

## 1. VIP 만족도(Stakeholder Satisfaction) 시스템

### 1-1. 핵심 규칙

| 규칙 | 내용 |
|------|------|
| 초기값 | 모든 VIP 만족도 **50/100** |
| 변동 폭 | 의사결정 1건당 **+5~+15 / -5~-15** |
| Phase 진행 조건 | 지정 VIP 만족도가 임계값 이상이어야 다음 Phase 진행 |
| 트레이드오프 | 대부분 **한 VIP ↑ 다른 VIP ↓** → 정답 없음 |
| 위험 구간 | **30 이하** 시 긴급 이벤트 발동 |
| 정보 기반 보너스 | 데이터로 근거를 제시하면 반대 VIP의 하락 폭 -3~-5 감소 |

### 1-2. VIP 등장 규칙

| 규칙 | 내용 |
|------|------|
| 동시 등장 금지 | 한 Phase에 VIP 1명만 활성화 |
| 활성/비활성 | 활성 VIP는 상세 정보 표시, 비활성 VIP는 역할만 표시 (접어두기) |
| 등장 배분 | VIP1: P1, P6 / VIP2: P4, P5 / VIP3: P2, P3 |

### 1-3. 엔딩 유형

| 엔딩 | 조건 | 설명 |
|------|------|------|
| 🏆 **균형 성장** | 3명 모두 60 이상 | 모든 이해관계자를 설득한 지속가능 전략 |
| ⚡ **편향 성장** | 1명 80+ but 1명 40 이하 | 한 축 올인 고위험 전략 |
| 🔥 **위기 봉합** | 2명 이상 40 이하 | 급한 불은 껐으나 구조적 문제 미해결 |

## 3. Phase별 상세 설계

### Phase 0: 게임 시작 (5분)

팀명, 팀원(유형 선택), 도메인 선택 후 챗봇에 등록.

---

### Phase 1: 문제 감지 — S-C-Q-A (15분)

브리핑 제시 후 팀이 S(Situation), C(Complication), Q(Question)를 분리·정의.

- 사후 라벨링: McKinsey S-C-Q-A
- VIP 3인 초기 코멘트 전달
- "문제 탐색 시작"으로 P2 전환

---

### Phase 2: 문제 탐색 — 질문·데이터 요청·VIP 인터뷰 (15분)

**P1에서 파악한 문제를 구체적으로 알기 위한 탐색 단계.**

팀이 자유롭게 질문하고 데이터를 요청. VIP에게 직접 질문 가능. 숨겨진 데이터 1~2개씩 점진 공개. 최소 3회 질문 후 "핵심 문제 정의서" 제출.

제출 양식:
```
[Phase 2: 문제 탐색 결과]
핵심 문제 (1문장): ___
발견한 단서:
1. ___
2. ___
3. ___
아직 모르는 것: ___
```

---

### Phase 3: 문제 분해 — MECE + Logic Tree (15분)

후행문제 1개 + 선행문제 3~5개로 구조 분해. 개인 LLM으로 다양한 절단면 탐색 후 팀 통합.

---

### Phase 4: 원인 추적 — 5-Whys + 성장 프레임워크 (20분)

#### 성장 프레임워크 선택

Phase 4 시작 시, 팀이 아래 3가지 중 하나를 선택:

| 프레임워크 | 관점 | 적합한 상황 |
|-----------|------|------------|
| **AARRR** (해적 지표) | Acquisition → Activation → Retention → Revenue → Referral | 퍼널 단계별 병목 분석 |
| **RARRA** (리텐션 우선) | Retention → Activation → Referral → Revenue → Acquisition | 리텐션이 핵심인 서비스 |
| **Growth Loop** (성장 루프) | Input → Action → Output → 재투입 | 바이럴/콘텐츠/유료 루프 순환 구조 분석 |

선택한 프레임워크로 5-Whys를 진행하며 근본 원인과 병목 단계를 진단.

제출 양식:
```
[Phase 4: 원인 추적]
선택 프레임워크: [AARRR / RARRA / Growth Loop]
5-Whys 체인: 1~5
근본 원인 (1문장): ___
병목 단계: ___
현재 상태: ___
```

---

### Phase 5: 가설 수립·검증 — If-Then-Because (25분)

2~3회 루프 반복. If-Then-Because 가설 제출 → 시뮬레이션 결과 반환.

---

### Phase 6: 종합 (15분)

전략서 제출 + 엔딩 확정 후 아래를 순서대로 출력:

1. **1-Pager PRD** (Phase 1~6 산출물 종합)
2. **PPT 생성 프롬프트** (GenSpark/Gamma에 복사-붙여넣기용)
3. **결과 시뮬레이션** — 팀의 활동 내용을 반영한 6개월 후 시뮬레이션
4. **엔딩 메시지** (균형/편향/위기)
5. **전환 브릿지 메시지**

#### 결과 시뮬레이션 형식

```
🔮 6개월 후 시뮬레이션: [서비스명]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

📊 핵심 지표 변화
| 지표 | 현재 | 3개월 후 | 6개월 후 |
|------|------|---------|---------|

👥 이해관계자 반응
- [VIP별 만족도 변동 + 코멘트]

🎯 팀 전략의 성과
▸ 잘한 점 / 아쉬운 점 / 예상 밖 변수

💡 실무 인사이트
```

**피칭은 P6 이후 오프라인에서 진행합니다. 게임 내 별도 단계 없음.**

### 6-2. UI 구성

- **좌측 사이드바 (1/3)**: 핵심 이해관계자 (VIP) + 상황 데이터 + 자세히 보기(3개월 추이 차트 모달)
- **우측 채팅 영역 (2/3)**: Phase 진행 바 + 메시지 + 입력창 (기본 3줄)
- **VIP 표시**: 활성 VIP는 1:1 아바타(108px) + 이름/역할/관심사/만족도바, 비활성 VIP는 역할만 표시
- **VIP 모달**: 클릭 시 첫 마디, 관심사, 만족도 상승/하락 요인, 등장 Phase 표시
- **상황 데이터 모달**: recharts 선 그래프로 4시점(3개월전→현재) 표시, 3개월 전 대비 증감률
- **GNB**: `fixed top-0 z-50`, 40px 높이
