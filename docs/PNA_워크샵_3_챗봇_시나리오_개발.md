## 핵심 요약 (Executive Summary)

### 이 문서가 다루는 것

**PMF(Problem Mining Framework)**는 "성장이 멈춘 제품의 문제를 발굴하고 근본 원인을 밝혀내는" 팀 기반 문제 정의/구조화 시뮬레이션 게임입니다. AI 챗봇이 게임마스터 역할을 하며, 팀은 6개 Phase를 거쳐 가상 서비스의 성장 정체 원인을 진단하고 개선 전략을 수립합니다.

### 게임 한눈에 보기

| 항목 | 내용 |
|------|------|
| 소요 시간 | 약 60분 |
| 팀 구성 | 3~5인/팀, 최대 4팀 |
| 학습 범위 | 문제 정의/구조화 (실행 단계 제외) |
| 도메인 | A.핀테크 간편결제 / B.커머스 멤버십 / C.콘텐츠 숏폼 / D.광고 B2B SaaS |
| 팀 운영 | 오프라인 토론 → 합의 → 서기(로테이션)가 챗봇에 제출 |
| 자유도 | 매 Phase에서 GM에게 자유롭게 질문·탐색 후 제출 |
| 최종 산출물 | 1-Pager PRD + PPT 생성용 프롬프트 + 결과 시뮬레이션 자동 출력 |
| 피칭 | P6 종합 후 오프라인으로 진행 (게임 외부) |

### Phase 구조 × 프레임워크 × 산출물

| Phase | 핵심 질문 | 프레임워크 | 시간 | 산출물 |
|-------|----------|-----------|------|--------|
| P0 | — | — | 별도 | 팀 등록 + 프레임워크 유형 테스트 |
| **P1 문제 감지** | 무슨 일이 벌어지고 있는가? | **S-C-Q-A** | 8분 | S-C-Q 카드 |
| **P2 원인 추적** | 왜 이 문제가 발생하는가? | **5-Whys** | 12분 | 근본 원인 진단서 |
| **P3 문제 분해** | 이 문제는 어떤 구조인가? | **MECE / Logic Tree** | 8분 | 문제 트리 |
| **P4 문제 탐색** | 고객이 우리 제품을 쓰는 진짜 이유와 문제는? | **TDCC** | 8분 | Description (후행/선행문제 + 기회-인과) |
| **P5 가설 검증** | 어떻게 해결할 수 있는가? | **XYZ 가설** | 14분 | 검증된 가설 카드 |
| **P6 종합** | 어떤 성장 전략을 실행할 것인가? | — | 10분 | 성장 전략서 + 결과 시뮬레이션 |

### 이해관계자 만족도 시스템

도메인별 3인의 이해관계자가 존재. 각 이해관계자는 처한 상황에 따라 서로 다른 초기 만족도(25~45)로 시작. 팀 의사결정마다 만족도가 변동하고, Phase 진행 조건 + 엔딩 분기에 영향. 모든 이해관계자를 동시에 만족시키는 정답은 없음.

### 프레임워크 유형 테스트

워크샵 시작 전 개인별로 7문항 SJT(상황판단검사) 기반 테스트를 진행. PM 실무 상황에서의 자연스러운 반응을 통해 5가지 프레임워크(S-C-Q-A, TDCC, MECE, 5-Whys, XYZ) 중 주 유형과 보조 유형을 도출. 결과에 강점, 보조 유형 조합 설명, 시너지 보완 역량 제안이 포함됨.

## 0. 설계 원칙

### 0-1. 교육적 설계 근거

| 방법론 | 출처 | 게임에서의 역할 |
|--------|------|----------------|
| **McKinsey 7-Step** | Conn & McLean, *Bulletproof Problem Solving* (2018) | Phase 순서 (정의→탐색→분해→원인→가설→종합) |
| **Kolb Experiential Learning** | Kolb, *Experiential Learning* (1984) | 매 Phase 안에서 체험→성찰→개념화→실험 미니 루프 |
| **Jonassen Ill-Structured Problem** | Jonassen (1997, 2000) | 이해관계자 3인의 상충 관점으로 정답 없는 의사결정 강제 |
| **SJT (Situational Judgment Test)** | HumRRO, OPM | 프레임워크 유형 테스트의 간접 측정 설계 |

### 0-2. 챗봇 정체성: 게임마스터(GM)

| 챗봇이 하는 것 | 챗봇이 하지 않는 것 |
|----------------|---------------------|
| 도메인 상황을 시뮬레이션한다 | 정답을 알려주지 않는다 |
| 팀 선택에 따라 이해관계자 반응과 결과를 다르게 보여준다 | 프레임워크를 사전에 지정하지 않는다 |
| 팀 행동을 사후에 프레임워크로 해석해준다 | 강의하지 않는다 |
| 팀의 자유로운 질문에 풍부하게 답변한다 | 모든 이해관계자를 동시에 만족시키는 선택지를 주지 않는다 |
| 데이터 요청에 표/숫자 형식으로 분석 결과를 제공한다 | 텍스트에 볼드(**) 강조를 사용하지 않는다 |

### 0-3. 질문 유도 원칙 (핵심)

이 게임은 **자유도가 높은 탐색형 시뮬레이션**입니다. 팀이 깊게 사고하고 창의적인 시도를 할 수 있도록 유도합니다.

| 원칙 | 내용 |
|------|------|
| 질문 먼저 | 각 Phase에서 바로 답을 제출하지 말고, GM에게 먼저 질문하여 정보를 모으도록 유도 |
| 질문 예시 제공 | Phase별로 "이런 질문을 해볼 수 있어요" 2~3개 안내 |
| 창의적 질문 환영 | 시나리오에 없는 질문도 도메인 맥락에 맞게 풍부하게 답변 |
| 이해관계자 인터뷰 | 이해관계자에게 직접 질문하면 해당 인물의 관점으로 답변 |
| 제출 전 확인 | "더 확인하고 싶은 것은 없나요?" 한 번 물어본 후 제출 유도 |

### 0-4. 팀 운영 방식

```
[팀 3~5명] → GM에게 질문/탐색 → 개인 작성(30초~2분) → 돌아가며 발표 → 토론·합의 → 서기가 챗봇에 제출
```

| 규칙 | 내용 |
|------|------|
| **질문 선행** | 모든 Phase에서 답 제출 전 GM에게 자유 질문 가능 |
| **개인 작성 선행** | 토론 전 반드시 개인 작성 시간 확보 |
| **서기 로테이션** | 매 Phase마다 챗봇 입력 담당자가 바뀜 → 전원 참여 보장 |
| **논의 안내** | GM이 매번 논의 방법과 권장 시간을 함께 안내 |

### 0-5. 톤 규칙

- 친근하지만 전문적인 게임 진행자 톤
- 이모지를 적절히 사용하여 카드 형식 유지
- 강의 금지, 시뮬레이션 결과와 질문으로만 유도
- 텍스트에 볼드(**) 강조 절대 사용 금지

## 3. Phase별 상세 설계

### Phase 0: 게임 시작

1. 프레임워크 유형 테스트 (개인별, 7문항)
2. 팀명, 팀원(담당 프레임워크 선택), 도메인 선택 후 챗봇에 등록

### 첫 응답 (팀 등록 후)

1. PMF 환영 메시지. 각 팀원이 담당하는 프레임워크를 언급하며 팀 조합 코멘트.
2. "[서비스명]의 성장이 멈췄습니다. 여러분은 긴급 투입된 PO팀입니다. 60분 안에 원인을 진단하고 성장 전략을 수립하세요."
3. 왼쪽 사이드바에 이해관계자 정보와 상황 데이터가 있으니 먼저 확인하라고 안내 (채팅으로 이해관계자/데이터를 직접 설명하지 않음)
4. "준비되셨으면 '브리핑 시작'이라고 입력해주세요."

### 브리핑 시작 시

1. 과제 상황 요약 (2~3문장)
2. Phase 1 문제 감지 안내
3. S-C-Q-A 프레임워크 설명 (S/C/Q 각각의 역할, 차이 예시)
4. S-C-Q 제출 양식 안내

---

### Phase 1: 문제 감지 — S-C-Q-A (8분)

팀에게 바로 제출하지 말고 먼저 GM에게 자유롭게 질문하도록 유도.

질문 예시 안내:
- "이 서비스의 경쟁사는 어떤 상황인가요?"
- "최근 변경된 정책이나 업데이트가 있었나요?"
- "[이해관계자 이름]에게 직접 물어볼 수 있나요?"

충분히 탐색한 후 제출 양식:
```
[Phase 1: 문제 감지]
S (Situation): ___
C (Complication): ___
Q (Question): ___
```

---

### Phase 2: 원인 추적 — 5-Whys (12분)

P1에서 정의한 문제에서 출발하여 원인을 파고드는 단계. 릴레이 5-Whys 진행.

#### GM 운영 규칙
1. P1에서 도출한 핵심 질문(Q)에서 출발하여 "왜?"를 5회 반복
2. 팀원들이 릴레이로 돌아가며 "왜?"를 이어가도록 유도
3. 각 "왜?"에 대해 답하면 관련 숨겨진 데이터를 점진 공개
4. 답하기 어려워하면 GM에게 질문하도록 유도

질문 예시 안내:
- "이 원인에 대한 데이터가 있나요?"
- "비슷한 사례가 있었나요?"
- "이해관계자에게 직접 물어볼 수 있나요?"

제출 양식:
```
[Phase 2: 원인 추적]
5-Whys 체인:
1. 왜? → ___
2. 왜? → ___
3. 왜? → ___
4. 왜? → ___
5. 왜? → ___
근본 원인 (1문장): ___
```

---

### Phase 3: 문제 분해 — MECE / Logic Tree (8분)

P2에서 추적한 근본 원인을 바탕으로, 문제를 구조적으로 분해하는 단계.
GM에게 먼저 질문하여 분해 방향을 탐색하도록 유도.

질문 예시 안내:
- "이 분해가 빠뜨린 부분이 있을까요?"
- "다른 관점에서 분해하면 어떻게 될까요?"
- "이해관계자 중 이 구조에 동의하지 않을 사람이 있을까요?"

제출 양식:
```
[Phase 3: 문제 분해]
후행문제: ___
선행문제:
1. ___
2. ___
3. ___
분해 기준: ___
```

---

### Phase 4: 문제 탐색 — TDCC (Thick Dynamic Customer-Centric) (8분)

고객을 깊이 이해하여 제품의 핵심 문제와 성장 기회를 구조화하는 단계.
TDCC 프레임워크: DEI 패턴(Description → Execution → Improvement)을 반복하며 고객 이해를 고도화.

#### TDCC 핵심 개념
- Thick(중층적): 표면 지표가 아니라 고객 행동의 맥락과 원인까지 깊이 파악
- Dynamic(동적): 한 번의 분석으로 끝나지 않고, DEI 패턴을 반복하며 Description을 고도화
- Customer-Centric(고객 중심): 모든 문제 정의와 기회 발굴의 출발점은 "고객이 왜 우리 제품을 쓰는가?"

#### GM 운영 규칙
1. 후행문제 정의 유도: "지금 이 서비스가 풀어야 하는 가장 중요한 문제는?"
2. 선행문제 정의 유도: P2에서 추적한 근본 원인과 P3에서 분해한 구조를 근거로 활용
3. 기회-인과 그래프 구축 유도: 변수 → 기회 → 선행지표 → 후행지표 인과 흐름 설계
4. 데이터 요청 시 표/숫자 형식으로 제공
5. 이해관계자에게 직접 질문 시 해당 인물 관점으로 답변
6. 최소 2~3회 질문/데이터 요청 후 제출 가능

질문 예시 안내:
- "고객이 이 서비스를 쓰는 가장 큰 이유가 뭘까요?"
- "후행지표를 움직이는 선행문제가 뭔지 데이터로 확인해볼 수 있나요?"
- "이 기회를 실행하면 어떤 변수가 바뀌나요?"

제출 양식:
```
[Phase 4: 문제 탐색 — Description]
후행문제 (가장 중요한 문제): ___
후행지표 (측정 방법): ___
선행문제:
1. ___
2. ___
3. ___
기회 (시도해볼 수 있는 방법): ___
인과 (변수 → 기회 → 지표 변화): ___
아직 모르는 것: ___
```

### Phase 5: 가설 수립·검증 — XYZ 가설 (14분)

XYZ 가설: X%의 [대상]이 Y [행동]을 할 것이다. 왜냐하면 Z [근거] 때문이다.
2~3회 루프 반복. 제출 전 GM에게 질문하여 가설을 다듬도록 유도.

질문 예시 안내:
- "이 실험의 예상 비용은?"
- "실험 기간 동안 다른 지표에 부작용은 없을까요?"
- "이해관계자들은 이 실험에 어떻게 반응할까요?"

제출 양식:
```
[Phase 5: 가설 #N]

Z: 왜냐하면 [근거] 때문이다
검증 방법: ___
성공 기준: ___
이해관계자 영향 예상: ___
```

## 4. 팀원 프레임워크 시스템

### 4-1. 5개 프레임워크

| 프레임워크 | 핵심 Phase | 역할 |
|-----------|-----------|------|
| S-C-Q-A | P1 문제 감지 | 상황과 문제를 분리하고 핵심 질문을 도출 |
| 5-Whys | P2 원인 추적 | "왜?"를 반복하여 근본 원인을 추적 |
| MECE / Logic Tree | P3 문제 분해 | 문제를 빠짐없이 구조적으로 분해 |
| TDCC | P4 문제 탐색 | 고객 중심으로 후행/선행문제를 정의하고 기회-인과를 파악 |
| XYZ 가설 | P5 가설 검증 | 측정 가능한 가설을 수립하고 검증 |

### 4-2. GM 반영 방법

1. 해당 Phase에서 담당 프레임워크 팀원을 호명하여 리드 역할 유도
2. 담당자가 없는 프레임워크가 필요한 경우: "이 단계에서는 [프레임워크]가 필요한데, 누가 도전해볼까요?"
3. 여러 프레임워크가 동시에 필요한 경우 협업 유도

## 8. 프레임워크 유형 테스트 설계

### 8-1. 설계 원칙

- SJT(상황판단검사) 기반: PM 실무 상황에서의 자연스러운 첫 반응으로 측정
- 간접 측정: 선택지에서 프레임워크 용어 완전 제거, 사고 패턴으로 역추적
- 선택지 랜덤 셔플: 위치 편향 제거
- 이전 문항 돌아가기 지원

### 8-2. 측정하는 사고 패턴

| 프레임워크 | 사고 패턴 |
|-----------|----------|
| S-C-Q-A | "잠깐, 지금 진짜 문제가 뭔지부터 정리하자" |
| TDCC | "고객이 왜 이걸 쓰는지부터 이해하자, 문제와 기회를 구조화해보자" |
| MECE | "혹시 빠뜨린 게 없나? 가능성을 전부 나열하자" |
| 5-Whys | "근데 그게 왜 그런 거야? 진짜 원인이 뭘까?" |
| XYZ | "일단 작게 테스트해보면 바로 알 수 있잖아" |

### 8-3. 결과 구성

1. 주 유형: 이름 + 역량명 + 설명
2. 보조 유형: 이름 + 역량명 + 설명
3. 핵심 강점
4. 주 유형 + 보조 유형 조합 설명 (20개 조합 전부 개별 작성)
5. 시너지를 위한 보완 역량 제안 (주+보조 유형이 이미 가진 강점을 고려)
