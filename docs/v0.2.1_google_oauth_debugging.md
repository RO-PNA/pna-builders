# v0.2.1 — Google OAuth Debugging & Hardening

> **Version**: v0.2.1 / **Status**: P0 코드 작업 완료 · 사용자 설정 대기 중 / **Priority**: Blocker for v0.3 admin UI
> **Test Date**: 2026-04-16 / **Environment**: localhost:3000 (dev)

v0.2 로그인은 **동작은 하지만 세션 내구성·서버 동기화·권한 분리에 구조적 결함**이 있다. 이 문서는 실제 테스트로 확인한 결함을 기록하고, v0.3(관리자 UI)가 올라가기 전에 반드시 정비해야 할 작업을 정의한다.

---

## 1. TL;DR

- ✅ Google 로그인 자체는 성공한다 (로그인 → 쿠키 세팅 → submit 가능)
- ⚠️ Supabase가 **Authorization Code (PKCE) 플로우**를 쓰는데, 앱에는 `/auth/callback` 라우트가 없다 → 클라이언트 하이드레이션에 의존 (불안정)
- ⚠️ `middleware.ts` 없음 → 서버 요청 간 세션 자동 갱신 불가
- ⚠️ 서버 Supabase 클라이언트가 쿠키 **`get`만** 구현 → 서버에서 세션 갱신해도 쿠키 write 안 됨 → stale 세션 위험
- ⚠️ `redirectTo: location.origin` 하드코딩 → 로그인 진입 페이지 유실
- ⚠️ `profiles` 테이블·관리자 플래그 없음 → 권한 분리 불가 (v0.3 블로커)

## 4. 결함 분석

### 4-1. `/auth/callback` 라우트 부재 ⚠️

**현재 플로우**
1. 유저가 "login" → `signInWithOAuth({ provider: 'google', redirectTo: location.origin })`
2. Supabase → Google 동의 → Supabase 콜백(`/auth/v1/callback`) → `redirect_to=http://localhost:3000?code=<code>&state=<state>`
3. 브라우저는 `/`로 진입 (**쿼리스트링에 `code` 포함**)
4. 클라이언트 하이드레이션 시 `@supabase/ssr`의 `createBrowserClient`가 URL에서 code를 감지하고 `exchangeCodeForSession`으로 세션 교환, 쿠키에 저장

**문제**
- 코드 교환이 **클라이언트 JS 로딩 성공에 의존** → JS 로딩 실패·봇·프리렌더 환경에서 불완전
- `?code=...&state=...`가 붙은 URL이 그대로 브라우저 히스토리에 남음
- 서버 컴포넌트가 첫 렌더할 때는 아직 세션 쿠키가 없어 **비로그인 상태로 렌더** → 하이드레이션 후 플래시

**권장**
- Next.js App Router에 전용 콜백 라우트 `src/app/auth/callback/route.ts` 추가
- 서버에서 `exchangeCodeForSession(code)` → 쿠키 세팅 → 원래 페이지로 `NextResponse.redirect`
- 이 방식이 공식 권장 (Supabase docs — "Server-Side Auth for Next.js")

### 4-2. `middleware.ts` 부재 ⚠️

**문제**
- 액세스 토큰 만료 시 서버 요청이 **stale 토큰**으로 처리됨
- 새로고침/네비게이션마다 `getUser()`가 최신 세션을 보장하지 않음

**권장**
- 루트 또는 `src/middleware.ts` 생성
- 모든 요청에서 `supabase.auth.getUser()` 호출로 세션 갱신 + 쿠키 write
- matcher로 `_next/static`, 이미지 등 제외

### 4-3. 서버 클라이언트 쿠키 핸들러 불완전 ⚠️

**현재 코드** (`src/utils/supabase/server.ts`)
```ts
cookies: {
  get(name) { return cookieStore.get(name)?.value },
  // set, remove 없음
}
```

**문제**
- 서버에서 토큰을 refresh해도 **새 쿠키를 Set-Cookie로 내려보낼 방법이 없음**
- `@supabase/ssr` 공식 패턴은 `getAll`/`setAll` 또는 `get`/`set`/`remove` 세트 구현을 요구

**권장** (App Router)
```ts
import { cookies } from 'next/headers'
import { createServerClient } from '@supabase/ssr'

export async function createSupabaseServer() {
  const cookieStore = await cookies()
  return createServerClient(URL, KEY, {
    cookies: {
      getAll() { return cookieStore.getAll() },
      setAll(cookiesToSet) {
        try {
          cookiesToSet.forEach(({ name, value, options }) =>
            cookieStore.set(name, value, options))
        } catch {
          // Server Component에서 호출되면 write가 실패할 수 있음 — 미들웨어에서 처리
        }
      },
    },
  })
}
```

### 4-4. `redirectTo: location.origin` 하드코딩 ⚠️

**문제**
- 유저가 `/submit`에서 로그인 유도되어도 로그인 후 무조건 `/`로 돌아감
- 워크샵·아이템 상세에서 로그인 유도 시 동일 문제

**권장**
- `signInWithOAuth({ redirectTo: `${origin}/auth/callback?next=${currentPath}` })`
- 콜백 라우트가 `next` 쿼리를 읽어 최종 리다이렉트

### 4-5. 사용자 프로필 동기화 없음 ⚠️

**현재**
- 네브바에 `user.email.split('@')[0]`만 표시
- 아바타·표시 이름·권한 정보 없음
- Google로부터 받은 `user_metadata.full_name`, `avatar_url` 미사용

**권장**
- `profiles` 테이블 생성 (`id uuid pk references auth.users(id)`, `display_name`, `avatar_url`, `is_admin boolean default false`, `created_at`)
- Supabase DB 트리거: `auth.users` insert → `profiles` insert (`raw_user_meta_data`에서 이름·아바타 추출)
- RLS: 공개 읽기(display_name/avatar_url만), 본인만 update
- v0.3 온톨로지 관리자 UI의 권한 기반 (= v0.2.1의 **핵심 목표**)

### 4-6. 기타 관찰
- 홈·상세·about·open-chat은 비로그인에서도 전체 열람 가능 — 의도된 설계
- `/api/comments` POST가 Supabase 세션에 의존한다면 동일 결함의 영향을 받음 (추가 검토 필요)
- 워크샵 챗봇(`/26-1q-workshop`)은 익명 세션을 생성 — 로그인 불필요 상태 유지

도메인 확정 후 별도 커밋으로 정리:
- `src/app/item/[id]/page.tsx` — `generateMetadata`의 OG URL `https://pna-builders.vercel.app/item/${id}` → 새 도메인
- `src/app/layout.tsx` — `metadata.metadataBase` 추가 (절대 URL 기준점)
- `README.md` — 프로덕션 URL 갱신
- `docs/*` — 프로덕션 URL 언급 갱신
