# PNA 오프라인 워크샵 — ① 구성 · 진행 가이드

> **문서 구분**: 본 문서는 3개 분리 문서 중 **①번 — 워크샵 구성 및 진행** 문서입니다.
> - ① 워크샵 구성 · 진행 (본 문서)
> - ② 프레임워크 매칭 체계 — 9개 프레임워크, 진단, 프롬프트 템플릿
> - ③ 챗봇 시나리오 · 개발 — 도메인 데이터, 갈등 카탈로그, 구현 가이드

---

## 1. 워크샵 개요

### 1-1. 기본 정보

| 항목 | 내용 |
|------|------|
| 참가자 | 실무 3-5년차 7-8명 + 팀장급(15년+) 3-4명, 총 10-12명 |
| 팀 구성 | 도메인별 3-4팀 (팀당 3-4명) |
| 소요시간 | 약 3시간 (180분) |
| 도메인 | 4개 도메인 중 당일 참가자에 맞춰 3-4개 운용 |
| 장소 | 팀별 테이블, 와이파이, 빔프로젝터/모니터 |

### 1-2. 워크샵 컨셉

> **"프레임워크를 강의로 배우지 않고, 게임처럼 체험한다"**

참가자들은 가상 도메인의 PO가 되어, 챗봇 게임마스터(GM)가 주는 미션을 6단계에 걸쳐 해결합니다. 팀이 자연스럽게 선택한 행동을 챗봇이 사후에 프레임워크로 해석해주는 구조입니다.

---

## 2. 핵심 설계 원칙

**① 사후 라벨링 (Post-hoc Framework Labeling)**
팀이 자연스럽게 선택한 행동을 사후에 "방금 여러분이 한 접근은 ___와 유사합니다"로 연결. 프레임워크를 사전에 지정/배정하지 않는다.

**② 턴제 시뮬레이션**
매 미션 2-3턴의 "행동→결과→후속 행동→결과" 루프. 팀의 선택에 따라 상황이 변함.

**③ 기대-현실 비교**
팀은 행동 전에 "기대하는 결과"를 적고, 챗봇이 시뮬레이션 결과와 비교. 이 과정 자체가 가설-검증 훈련.

**④ 예상 밖 요소**
매 턴 결과에 반드시 1개 이상의 "팀이 예상하지 못한 발견" 포함.

**⑤ 개인 관점 → 팀 합의**
개인이 각자의 프레임워크 렌즈로 먼저 분석하고, 서로 다른 관점을 팀 논의에서 합치는 과정을 반복.

## 3. 이중 AI 구조: 팀 챗봇 + 개인 LLM

본 워크샵에는 두 가지 AI 활용 레이어가 있습니다.

| 레이어 | 도구 | 역할 |
|--------|------|------|
| **팀 챗봇** | 워크샵 전용 챗봇 (Claude API) | 게임마스터. 상황 시뮬레이션, 미션 진행, 사후 프레임워크 라벨링 |
| **개인 LLM** | ChatGPT/Claude 등 각자 사용하는 AI | 아이스브레이킹에서 지정받은 프레임워크 관점으로 문제를 분석하는 개인 도구 |

매 턴 흐름:
```
[개인] 각자의 LLM + 프레임워크 프롬프트로 개인 분석 (3분)
  ↓
[팀] 개인 분석을 공유하고 팀 액션 카드로 합의 (5분)
  ↓
[팀 챗봇] 팀 액션 카드 제출 → 결과 시뮬레이션
```

---

## 4. 전체 흐름

```
[Phase 0] 아이스브레이킹 — 구글폼 (15분)
  → 개인별 문제해결 성향 진단 + 프레임워크 매칭
  → 개인 LLM용 프레임워크 프롬프트 템플릿 배포

[Phase 1] 팀 등록 + 도메인 브리핑 — 챗봇 (15분)
  → 팀 구성 + 성향 조합 입력
  → 도메인 선택 + 초기 데이터 수령

[Phase 2] 팀 미션 체인 — 챗봇 + 개인 LLM (120분)
  → 6개 미션 × 2-3턴 턴제 시뮬레이션
  → 매 턴: 개인 LLM 분석 → 팀 합의 → 챗봇 제출 → 결과

[Phase 3] 피칭 + 크로스 피드백 (25분)

[Phase 4] 클로징 — 챗봇 (10분)
  → 팀 프레임워크 누적 리뷰 + 개인별 활용 빈도 비교
```

## 5. Phase 0: 아이스브레이킹 (구글폼, 15분)

### 5-1. 진행 방식

| 순서 | 활동 | 시간 |
|------|------|------|
| 1 | 구글폼 응답 (4개 상황 × 4지선다) | 5분 |
| 2 | 결과 공유 + 자기 유형 소개 | 5분 |
| 3 | 프레임워크 프롬프트 템플릿 배포 | 5분 |

### 5-2. 구글폼 구성: 4개 상황 × 4지선다

참가자는 각 상황에서 **가장 먼저 하고 싶은 행동** 1개를 선택합니다.

---

**상황 1: "출시 2주 된 서비스인데 DAU가 매일 5%씩 빠지고 있습니다"**

| 선택지 | 유형 |
|--------|------|
| A) 최근 이탈한 유저 5명을 직접 만나서, 그들이 지금 대신 쓰는 서비스가 뭔지 물어본다 | 🔍 고객 발견 |
| B) DAU 하락을 퍼널 단계별로 쪼개서, 어느 구간에서 이탈이 집중되는지 로직트리를 그린다 | 📐 구조 분해 |
| C) "DAU가 빠진다"는 현상 뒤의 근본 원인을 5번 연속 "왜?"를 물어서 추적한다 | 🔗 원인 추적 |
| D) "만약 온보딩 3단계를 2단계로 줄이면 Day 3 리텐션이 15%→25%로 오를 것이다"라는 가설을 세우고 검증 방법을 설계한다 | 🧪 가설 실험 |

---

**상황 2: "경쟁사가 우리와 거의 같은 기능을 절반 가격에 출시했습니다"**

| 선택지 | 유형 |
|--------|------|
| A) 가격 때문에 이탈을 고민하는 고객을 찾아가서, 그들이 우리 서비스를 "어떤 상황에서, 어떤 목적으로" 쓰는지 관찰한다 | 🔍 고객 발견 |
| B) 고객이 서비스를 선택하는 기준을 MECE하게 (가격/기능/브랜드/전환비용/습관) 정리하고, 각 기준별 우리 vs 경쟁사 점수를 비교한다 | 📐 구조 분해 |
| C) "왜 경쟁사가 같은 기능을 절반 가격에 만들 수 있는가?" → "왜 우리의 원가 구조는 이렇게 되어 있는가?"로 파고 들어간다 | 🔗 원인 추적 |
| D) "가격이 아닌 ___가 전환의 핵심 변수다"라는 가설을 세우고, 2주 내 검증 가능한 실험을 설계한다 | 🧪 가설 실험 |

---

**상황 3: "CEO가 '다음 달까지 MAU 2배로 올려'라고 했습니다"**

| 선택지 | 유형 |
|--------|------|
| A) 지금 매일 들어오는 유저들이 "어떤 일을 해결하려고(Job)" 우리 서비스를 여는지 파악하고, 그 Job을 더 잘 수행하게 만드는 데 집중한다 | 🔍 고객 발견 |
| B) MAU를 "신규 유입 × 활성화율 × 리텐션 + 부활 유저"로 분해하고, 어디를 개선하는 게 2배 달성에 가장 효율적인지 시뮬레이션한다 | 📐 구조 분해 |
| C) "왜 지금 MAU가 이 수준에 머물러 있는가?"를 반복 질문하며, CC를 제한하는 근본 병목(선행문제)을 찾는다 | 🔗 원인 추적 |
| D) MAU 2배가 구조적으로 가능한지 CC 공식에 대입해보고, "만약 이탈률을 5%p 줄이면 CC가 ___가 된다"는 정량 가설로 전환한다 | 🧪 가설 실험 |

---

**상황 4: "AI를 도입하라는 요구가 있는데, 어디에 어떻게 써야 할지 모르겠습니다"**

| 선택지 | 유형 |
|--------|------|
| A) 고객이 서비스 안에서 가장 번거로워하는 순간을 관찰하고, AI가 그 "고용 맥락(Job Context)"에서 마찰을 줄일 수 있는지 탐색한다 | 🔍 고객 발견 |
| B) AI 적용 가능 영역을 (획득 / 활성화 / 리텐션 / 수익 / 추천)으로 MECE 분류하고, 각 영역의 임팩트와 실현 가능성을 매트릭스로 평가한다 | 📐 구조 분해 |
| C) "왜 리텐션이 낮은가?" → "왜 아하 모멘트에 도달하지 못하는가?" → "왜 핵심 행동을 안 하는가?" 순서로 AI가 개입할 인과 지점을 찾는다 | 🔗 원인 추적 |
| D) AI 적용 후보 3곳을 뽑고, 각각 "만약 ___에 AI를 적용하면, ___가 현재 X에서 Y로 변할 것이다"라는 XYZ 가설로 바꿔서 우선순위를 정한다 | 🧪 가설 실험 |

---

### 5-3. 성향 판정 및 프레임워크 매칭

가장 많이 선택한 유형 = **1순위 (Primary)**, 두 번째로 많은 유형 = **2순위 (Complementary)**

| 유형 | 별칭 | 배포 프레임워크 |
|------|------|----------------|
| 🔍 고객 발견형 | 현장 탐정 | IDEO 디자인씽킹 + JTBD |
| 📐 구조 분해형 | 구조 설계자 | MECE+LogicTree + CIRCLES + S-C-Q-A |
| 🔗 원인 추적형 | 인과 추적자 | 5-Whys + TDCC |
| 🧪 가설 실험형 | 속도전 실험가 | If-Then-Because + XYZ 가설 |

> **프레임워크 세부 내용, LLM 프롬프트 템플릿** → 📄 **② 프레임워크 매칭 체계** 문서 참조

### 5-4. 운영자 가이드 (아이스브레이킹)

- 결과를 화면에 띄우고, 각 참가자가 자기 유형을 소개하며 간단히 이야기를 나누도록 유도 (5분)
- 팀 편성 시 가능하면 **서로 다른 유형이 섞이도록** 배치
- 유형이 쏠려도 게임 진행에 지장 없음 (사후 라벨링 방식이므로)
- 각 참가자에게 **해당 유형의 프레임워크 요약 + LLM 프롬프트 템플릿** 배포 (인쇄물 또는 QR)
- 프레임워크가 2-3개인 유형은, 미션 초반(M1-M3)엔 Primary를, 후반(M4-M6)엔 Complementary를 쓰도록 안내

## 6. Phase 1: 팀 등록 + 도메인 브리핑 (챗봇, 15분)

### 6-1. 팀 등록

**[Step 1: 팀 이름]**
```
GM: 🎮 PNA 워크샵에 오신 것을 환영합니다!
    팀 이름을 정해주세요.
```

**[Step 2: 팀원 입력]**
```
GM: 팀원 정보를 알려주세요.
    이름과 아이스브레이킹 결과를 적어주세요.
    예: 민수 / 🔍 고객 발견형, 지현 / 📐 구조 분해형
```

**[Step 3: 팀 성향 요약]**
```
GM: 🧬 팀 구성을 확인했습니다.
    (팀원별 유형 나열)
    
    오늘 게임에서는 정해진 역할 없이 팀이 자유롭게 행동합니다.
    여러분이 한 행동을 제가 사후에 프레임워크로 해석해드릴게요.
    
    💡 개인 LLM 활용 가이드:
    매 턴마다 팀 논의 전에, 각자 배포받은 프레임워크 프롬프트로
    개인 분석을 먼저 해보세요. 서로 다른 관점이 모이면 
    팀의 의사결정이 더 풍부해집니다.
```

### 6-2. 도메인 선택

```
GM: 오늘 도전할 도메인을 선택하세요.

    [A] 🛒 패션 커머스 "스타일픽" — 리텐션이 안 나오는 커머스
    [B] 💼 B2B SaaS "태스크플로우" — 온보딩에서 사라지는 고객사
    [C] 📱 뉴스레터 "모닝브리프" — 구독자는 많은데 수익이 안 나는 미디어
    [D] 🏥 헬스케어 앱 "케어루틴" — 습관이 안 만들어지는 건강관리 앱
    
    [직접 입력] 우리 팀만의 도메인을 제안할 수도 있습니다.
```

> **도메인 상세 데이터, 숨겨진 데이터, 과제 목록** → 📄 **③ 챗봇 시나리오 · 개발** 문서 참조

## 7. Phase 2: 팀 미션 체인 (120분)

### 7-1. 턴제 게임 사이클

```
┌──────────────────────────────────────────────────┐
│  👤 개인 LLM 분석 (3분)                            │
│  각자의 프레임워크 프롬프트로 상황 분석             │
├──────────────────────────────────────────────────┤
│  🤝 팀 토론 (5분)                                   │
│  개인 분석을 공유하고 팀 액션 카드로 합의           │
├──────────────────────────────────────────────────┤
│  🎲 상황 카드 (챗봇)                                │
│  현재 상황 + 2-3개 행동 옵션 + 직접 입력            │
├──────────────────────────────────────────────────┤
│  📋 팀 액션 카드 (팀 → 챗봇 제출)                   │
│  선택한 행동 + 구체적 방법 + 기대하는 결과          │
├──────────────────────────────────────────────────┤
│  📊 결과 카드 (챗봇)                                │
│  시뮬레이션 결과 + 기대 vs 현실 비교                │
│  + 예상 밖 발견 + 🧠 프레임워크 사후 라벨링         │
│  + (선택) 갈등/딜레마/장애물 발동                    │
├──────────────────────────────────────────────────┤
│  → 다음 턴 또는 미션 종료                           │
└──────────────────────────────────────────────────┘
```

### 7-2. 팀 액션 카드 양식

**[간이 모드 — Mission 1~2]**

```
━━━━━━━━━━━━━━━━━━━━━━━━
📋 팀 액션 카드 (간이)
━━━━━━━━━━━━━━━━━━━━━━━━
1. 선택한 행동: [옵션 또는 직접 입력]
2. 기대하는 결과: "이렇게 하면 ___를 알 수 있을 것이다"
3. (선택) 첨부: [화이트보드 사진, 메모 등]
━━━━━━━━━━━━━━━━━━━━━━━━
```

**[전략 모드 — Mission 3~6]**

```
━━━━━━━━━━━━━━━━━━━━━━━━
📋 팀 액션 카드 (전략)
━━━━━━━━━━━━━━━━━━━━━━━━
1. 선택한 행동: [옵션 또는 직접 입력]

4. 이게 틀렸다면: "___가 나오면 이 가설은 틀린 것이다"
5. (선택) 첨부: [화이트보드 사진, 스케치 등]
```

## 11. 퍼실리테이터 운영 가이드

### 11-1. 사전 준비 체크리스트

| 항목 | 상세 |
|------|------|
| 구글폼 | 4개 상황 × 4지선다 폼 세팅, 자동 집계 확인 |
| 프레임워크 자료 | 유형별 프롬프트 템플릿 인쇄물 or QR 코드 준비 |
| 챗봇 | 도메인별 시스템 프롬프트 세팅, 팀 코드 발급 |
| 현장 | 팀별 테이블, 와이파이, 공유 화면, 타이머 앱 |
| 비상 | 챗봇 장애 시 퍼실리테이터가 수동으로 결과 카드 읽어주는 백업 플랜 |

### 11-2. 진행 중 핵심 원칙

**시간 관리**
- 각 턴의 제한시간을 엄격하게 관리 (타이머 사용)
- M1-M2는 워밍업이므로 약간 유연하게, M3 이후는 엄격하게
- 전체 진행이 밀리면 M4와 M5를 각 1턴으로 줄여서 조정

**에너지 관리**
- M2 후 반드시 10분 휴식 (에너지 급락 방지)
- 팀이 너무 순탄하게 진행하면 갈등/딜레마 카드를 추가 발동
- 팀이 이미 어려워하고 있다면 장애물을 줄이고 격려

**개입 최소화**
- 퍼실리테이터는 챗봇과 팀 사이의 "기술 지원"만 담당
- 팀의 전략적 판단에는 개입하지 않음
- "정답은 없습니다" — 이 메시지를 반복 강조

### 11-3. 비상 시나리오 대응

| 상황 | 대응 |
|------|------|
| 챗봇 응답 지연/장애 | 퍼실리테이터가 도메인 데이터를 보며 수동으로 결과 카드를 구두로 전달 |
| 팀이 미션에서 막힘 | 챗봇에 "힌트 요청" 기능 활성화. 힌트는 최대 2회 |
| 시간 초과 | M4/M5를 각 1턴으로 축소, 또는 M6를 피칭 준비 시간으로 흡수 |
| 참가자 불균형 | 2인 팀이 되더라도 게임 진행 가능. 개인 LLM을 더 적극 활용하도록 안내 |
