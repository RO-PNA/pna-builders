## 1. 유형 체계: 9개 프레임워크 → 4개 문제해결 유형

### 1-1. 클러스터링 근거

9개 프레임워크를 **"문제에 부딪혔을 때 가장 먼저 하는 행동"** 기준으로 분류합니다.

```
"고객을 먼저 본다"  →  고객 발견형
"구조를 먼저 짠다"  →  구조 분해형
"원인을 먼저 판다"  →  원인 추적형
"가설을 먼저 세운다" →  가설 실험형
```

### 1-2. 4개 유형 × 9개 프레임워크 매핑

| 유형 | 별칭 | 프레임워크 | 핵심 질문 |
|------|------|-----------|-----------|
| **A. 고객 발견형** | 🔍 현장 탐정 | ① IDEO 디자인씽킹 ② Jobs To Be Done | "고객은 지금 무엇을 경험하고 있는가?" |
| **B. 구조 분해형** | 📐 구조 설계자 | ③ MECE + Logic Tree ④ 구글 CIRCLES Method ⑤ 맥킨지 S-C-Q-A | "이 문제를 빠짐없이 쪼개면 핵심은 어디인가?" |
| **C. 원인 추적형** | 🔗 인과 추적자 | ⑥ 도요타 5-Whys ⑦ 토스 TDCC 프레임워크 | "왜? 왜? 왜? 근본 원인은 무엇인가?" |
| **D. 가설 실험형** | 🧪 속도전 실험가 | ⑧ If-Then-Because ⑨ XYZ 가설 | "검증 가능한 가설로 전환하면 어떻게 되는가?" |

### 1-3. 유형 간 관계

```
[A. 고객 발견] → 문제의 맥락과 니즈를 발견한다
      ↓
[B. 구조 분해] → 발견한 문제를 체계적으로 분해한다
      ↓
[C. 원인 추적] → 분해된 문제 중 근본 원인을 추적한다
      ↓
[D. 가설 실험] → 원인에 대한 가설을 세우고 검증한다
```

> 이 순서가 "정답 프로세스"라는 뜻이 아닙니다.
> 실무에서는 어디서든 시작할 수 있고, 왔다갔다 합니다.
> 핵심은 **4개 관점이 팀 안에서 모두 나오는 것**입니다.

## 2. 아이스브레이킹 진단 (구글폼)

### 2-1. 진단 구조

참가자는 4개 상황에서 각각 **가장 먼저 하고 싶은 행동** 1개를 선택합니다.
4개 응답 중 가장 많이 선택한 유형 = **1순위 (Primary)**, 두 번째 = **2순위 (Complementary)**.

> 구글폼 4개 상황 × 4지선다 상세 → 📄 **① 구성·진행** 문서 섹션 5 참조

### 2-2. 성향 판정 및 프레임워크 배포

| 유형 | 별칭 | 배포 프레임워크 | 개인 LLM 프롬프트 |
|------|------|----------------|------------------|
| 🔍 고객 발견형 | 현장 탐정 | IDEO 디자인씽킹 + JTBD | 프롬프트 A (섹션 3-1, 3-2) |
| 📐 구조 분해형 | 구조 설계자 | MECE+LogicTree + CIRCLES + S-C-Q-A | 프롬프트 B (섹션 3-3, 3-4, 3-5) |
| 🔗 원인 추적형 | 인과 추적자 | 5-Whys + TDCC | 프롬프트 C (섹션 3-6, 3-7) |
| 🧪 가설 실험형 | 속도전 실험가 | If-Then-Because + XYZ 가설 | 프롬프트 D (섹션 3-8, 3-9) |

### 2-3. 프레임워크 사용 순서

프레임워크가 2-3개인 유형은 미션 단계별로 나눠 사용합니다.

| 유형 | Primary (M1-M3) | Complementary (M4-M6) |
|------|-----------------|----------------------|
| 🔍 고객 발견형 | IDEO 디자인씽킹 | JTBD |
| 📐 구조 분해형 | MECE + Logic Tree | CIRCLES 또는 S-C-Q-A |
| 🔗 원인 추적형 | 5-Whys | TDCC (DEI 패턴) |
| 🧪 가설 실험형 | If-Then-Because | XYZ 가설 |

> 물론 상황에 따라 자유롭게 바꿔도 됩니다. 이 구분은 가이드라인일 뿐입니다.

### 3-2. Jobs To Be Done (JTBD)

#### 핵심 요약

| 항목 | 내용 |
|------|------|
| 출처 | Clayton Christensen / Bob Moesta |
| 한 문장 | 고객은 제품을 "구매"하는 것이 아니라, 특정 상황에서 특정 Job을 해결하기 위해 "고용(hire)"하는 것이다 |
| 핵심 구조 | **"[상황]에서 [Job]을 달성하기 위해 [제품]을 고용한다"** |
| JTBD 공식 | "When I ___, I want to ___, so I can ___" |
| 언제 강력한가 | 제품이 대체하는 것이 무엇인지, 경쟁의 본질이 어디인지 파악할 때 |
| 한계 | Job을 너무 추상적으로 정의하면 실행 가능한 인사이트가 안 나옴 |

#### 실무 적용 포인트

```
• 경쟁사는 같은 카테고리가 아님. 같은 Job을 해결하는 모든 대안이 경쟁.
  → 토스의 진짜 경쟁사: 다른 송금 앱이 아니라 "현금", "카카오톡 계좌 전달"

• Job에는 3가지 레이어:
  ① Functional Job: 실용적 과업 (송금을 빨리 하고 싶다)
  ② Emotional Job: 감정적 욕구 (바보 같은 느낌 안 받고 싶다)
  ③ Social Job: 사회적 맥락 (친구들 앞에서 센스 있어 보이고 싶다)

• "전환의 4가지 힘" (Switch Interview):
  Push: 현재 솔루션에 대한 불만
  Pull: 새 솔루션에 대한 매력
  Anxiety: 새 솔루션에 대한 불안
  Habit: 현재 솔루션에 대한 관성
```

### 3-3. MECE + Logic Tree

#### 핵심 요약

| 항목 | 내용 |
|------|------|
| 출처 | 맥킨지 (Barbara Minto) |
| 한 문장 | 문제를 "빠짐없이(CE), 겹침없이(ME)" 분해하여, 복잡한 상황에서 핵심 레버를 놓치지 않고 찾아내는 구조화 기법 |
| MECE 원칙 | Mutually Exclusive (상호 배타), Collectively Exhaustive (전체 포괄) |
| Logic Tree 구조 | 상위 문제 → 하위 요소 분해 → 각 요소별 크기/영향력 평가 → 핵심 레버 선택 |
| 언제 강력한가 | "어디서부터 손대야 할지 모르겠다"는 상황에서 전체를 조망하고 우선순위를 잡을 때 |
| 한계 | 깔끔한 분해에 집착하면 현실의 복잡한 상호작용을 놓칠 수 있음 |

#### 실무 적용 포인트

```
• CC 공식 자체가 Logic Tree의 좋은 예시:
  MAU = f(Inflow, Churn)
    Inflow = 오가닉 + 광고 + 바이럴 + 부활
    Churn = f(아하모멘트 달성률, 제품 가치, 경쟁 대안)

• 분해 → 사이징이 핵심:
  "어디가 큰가?"를 먼저 보고 가장 큰 레버에 집중
  → 예: 이탈의 57%가 Day 1-3에 집중 → 여기가 가장 큰 레버

• Issue Tree vs Hypothesis Tree:
  Issue Tree: 열린 질문으로 분해 ("무엇이 문제인가?")
  Hypothesis Tree: 가설로 분해 ("A가 문제일 것이다 / B가 문제일 것이다")
```

### 3-4. 구글 CIRCLES Method

#### 핵심 요약

| 항목 | 내용 |
|------|------|
| 출처 | Lewis C. Lin (구글 PM 인터뷰 기반) |
| 한 문장 | 제품 문제를 7개 체계적 단계로 접근하여 빠짐없이 분석하는 구조화된 문제해결 프레임워크 |
| 7단계 | **C**omprehend → **I**dentify → **R**eport → **C**ut → **L**ist → **E**valuate → **S**ummarize |
| 풀어쓰기 | 이해하기 → 유저 정의 → 니즈 나열 → 우선순위 결정 → 솔루션 나열 → 평가 → 요약 |
| 언제 강력한가 | "이 제품 어떻게 개선할래?" 같은 열린 질문에 체계적으로 답할 때 |
| 한계 | 순서가 경직되면 창의적 도약이 어려울 수 있음 |

#### 실무 적용 포인트

```
• C - Comprehend: 문제를 제대로 이해했는지 확인 질문 먼저
  → "리텐션 올려달라"는 요청을 받으면 "어떤 세그먼트의 리텐션?"부터 확인

• I - Identify: 가장 중요한 유저 세그먼트를 특정
  → 전체 유저가 아닌, 가장 임팩트가 큰 세그먼트에 집중

• R → C: 니즈를 나열한 뒤 반드시 우선순위 잘라내기(Cut)
  → 모든 니즈를 다 풀 수 없음. 임팩트 × 실현가능성으로 2-3개 선택

• E - Evaluate: 솔루션을 평가할 때 트레이드오프를 명시
  → "이 솔루션은 A에 강하지만 B를 희생한다"
```

#### 프롬프트 템플릿

```
당신은 구글 CIRCLES Method 전문가입니다.
아래 상황을 CIRCLES 7단계로 체계적으로 분석해주세요.

## 현재 상황
[챗봇 상황 카드 내용 붙여넣기]

## 분석 요청 (CIRCLES 7단계)

1. **C - Comprehend (이해)**
   이 상황의 핵심을 정확히 이해하기 위해 먼저 확인해야 할 
   질문 2-3개를 작성해주세요. (가정이 필요한 부분 명시)

2. **I - Identify Customer (유저 정의)**
   이 서비스의 유저를 세그먼트로 나누고, 
   지금 가장 집중해야 할 세그먼트 1개를 선택하고 이유를 설명.

3. **R - Report Needs (니즈 나열)**
   선택한 세그먼트의 니즈를 5개 이상 나열.

4. **C - Cut (우선순위)**
   나열한 니즈 중 상위 2개를 선택. 
   선택 기준: 임팩트(크기) × 실현가능성(비용/시간)

5. **L - List Solutions (솔루션 나열)**
   선택한 니즈를 해결하는 솔루션 3개를 제시.

6. **E - Evaluate (평가)**
   각 솔루션의 장점, 단점, 트레이드오프를 한 줄씩 정리하고
   최종 추천 1개를 선택.

7. **S - Summarize (요약)**
   "누구에게 / 어떤 문제를 / 어떻게 해결하면 / 어떤 지표가 변한다"를 
   2-3문장으로 요약.

+ **팀에 제안할 행동 1가지**

한국어로, 팀 토론에서 3분 안에 공유할 수 있는 분량으로 작성해주세요.

### 3-5. 맥킨지 S-C-Q-A (Situation-Complication-Question-Answer)

#### 핵심 요약

| 항목 | 내용 |
|------|------|
| 출처 | 맥킨지 / Barbara Minto (피라미드 원칙) |
| 한 문장 | 복잡한 문제를 "상황(S) → 복잡성(C) → 핵심 질문(Q) → 답(A)"의 스토리라인으로 정리하여, 논리적 설득력을 갖춘 커뮤니케이션을 만드는 구조 |
| 4단계 | **S**ituation (모두가 동의하는 현재 상황) → **C**omplication (그런데 문제가 있다) → **Q**uestion (그래서 우리가 답해야 할 질문은?) → **A**nswer (우리의 답은 이것이다) |
| 언제 강력한가 | CEO에게 보고할 때, 팀을 설득할 때, 피칭 구조를 잡을 때 |
| 한계 | 분석 도구가 아니라 커뮤니케이션 도구. 다른 분석과 병행 필요 |

#### 실무 적용 포인트

```
• S(상황)은 "청중이 이미 동의하는 사실"로 시작
  → "우리 서비스의 월 리텐션은 18%입니다" (데이터 기반 사실)

• C(복잡성)에서 긴장감을 만듦
  → "그런데 CEO는 AI 추천으로 해결하라 하지만, 데이터는 다른 원인을 가리킵니다"

• Q(질문)는 하나로 수렴
  → "그래서 우리는 AI를 추천에 쓸 것인가, 사이즈 문제에 쓸 것인가?"

• A(답)는 So What → Why So → So What 구조로 깊이를 더함

• 미션 6(피칭)에서 특히 강력한 프레임워크
```

```
당신은 맥킨지 S-C-Q-A 스토리라인 전문가입니다.
아래 상황을 S-C-Q-A 구조로 정리해주세요.

## 현재 상황
[챗봇 상황 카드 내용 붙여넣기]

## 분석 요청

1. **S - Situation (상황)**
   모두가 동의할 수 있는 현재 사실 2-3개를 나열해주세요.
   (의견이 아닌 데이터 기반 사실)

2. **C - Complication (복잡성)**
   "그런데..."로 시작하는 핵심 긴장 포인트를 정리해주세요.
   - 기대와 현실의 차이
   - 상충하는 요구사항
   - 숨겨진 위험

3. **Q - Question (핵심 질문)**
   위 복잡성에서 도출되는 "우리가 답해야 할 가장 중요한 질문"을

   - 근거 2: ___
   - 근거 3: ___

5. **팀에 제안할 행동 1가지**
   S-C-Q-A 관점에서 이 턴에 팀이 해야 할 가장 중요한 행동과 이유.
   (특히: 지금 부족한 것이 S인가? C인가? Q인가? A인가?)

한국어로, 팀 토론에서 3분 안에 공유할 수 있는 분량으로 작성해주세요.
```

### 3-6. 도요타 5-Whys

#### 핵심 요약

| 항목 | 내용 |
|------|------|
| 출처 | 도요타 생산 시스템 (Sakichi Toyoda) |
| 한 문장 | 표면적 현상에 대해 "왜?"를 5번 반복하여, 겉으로 보이는 증상이 아닌 근본 원인(Root Cause)에 도달하는 기법 |
| 핵심 원칙 | 모든 문제에는 보이지 않는 깊은 원인이 있다. 증상을 치료하면 문제가 재발한다. |
| 구조 | 현상 → Why 1 → Why 2 → Why 3 → Why 4 → Why 5 → 근본 원인 |
| 언제 강력한가 | "이미 해결했는데 또 같은 문제가 반복된다"는 상황에서 근본 원인을 찾을 때 |
| 한계 | 단선적 인과에 치우칠 수 있음. 복합 원인 상황에서는 분기가 필요 |

#### 실무 적용 포인트

```
• "5번"은 관습적 숫자. 근본 원인에 도달하면 3번이든 7번이든 상관없음

• 분기 지점 주의:
  Why 2에서 원인이 2개 이상일 수 있음
  → 각 분기를 따로 추적해야 함 (5-Whys Tree)

• "왜?"에 대한 답은 검증 가능해야 함
  → "고객이 관심이 없어서"는 답이 아님
  → "Day 3까지 위시리스트를 등록한 유저가 12%뿐이다"가 답

• 토스 사례: 왜 리텐션이 낮은가? → 왜 아하 모멘트에 못 도달하는가?
  → 왜 4일 내 2회 송금을 못 하는가? → 왜 계좌에 돈이 없는가?
  → 해결: 무료 송금 5회 제공
```

#### 프롬프트 템플릿

```
당신은 도요타 5-Whys 근본 원인 분석 전문가입니다.
아래 상황의 표면적 문제에서 근본 원인까지 파고 들어주세요.

## 현재 상황
[챗봇 상황 카드 내용 붙여넣기]

## 분석 요청

1. **표면 현상 정의**
   지금 관찰되는 가장 명확한 문제 현상을 한 문장으로.

2. **5-Whys 체인**
   Why 1: 왜 [표면 현상]이 발생하는가?
     → 답: ___
   Why 2: 왜 [Why 1의 답]인가?
     → 답: ___
   Why 3: 왜 [Why 2의 답]인가?
     → 답: ___
   Why 4: 왜 [Why 3의 답]인가?
     → 답: ___
   Why 5: 왜 [Why 4의 답]인가?
     → 답: ___ ← 이것이 근본 원인 후보

   * 분기가 생기면 각 분기를 별도로 표시해주세요.

3. **근본 원인 검증**
   도출된 근본 원인이 정말 맞는지 확인하려면 어떤 데이터를 봐야 하는가?

4. **증상 치료 vs 근본 치료 비교**
   - 증상 치료 (Why 1-2 수준에서 해결): 어떤 조치? 효과는? 한계는?
   - 근본 치료 (Why 4-5 수준에서 해결): 어떤 조치? 효과는? 비용은?

5. **팀에 제안할 행동 1가지**
   5-Whys 관점에서 이 턴에 팀이 해야 할 가장 중요한 행동과 이유.

### 3-7. 토스 TDCC 프레임워크 (DEI 패턴)

#### 핵심 요약

| 항목 | 내용 |
|------|------|
| 출처 | 토스 메이커스 컨퍼런스 2025 |
| 한 문장 | 문제를 후행/선행으로 분해하고, 기회와 변수 간 인과관계를 그래프로 설계하여 "왜 효과가 있는지"를 구조적으로 증명하는 접근법 |
| 핵심 패턴 | DEI: Description(문제 구조화) → Execution(실행) → Improvement(개선) 반복 |
| 핵심 도구 | 후행/선행문제 정의, 문제 트리, 기회-인과 그래프 |
| 언제 강력한가 | "이 솔루션이 왜 작동하는가?"를 변수 간 인과관계로 설명해야 할 때 |
| 한계 | 분석에 시간이 많이 걸릴 수 있고, 완벽한 인과 모델을 추구하다 실행이 늦어질 위험 |

#### 실무 적용 포인트

```
• 후행문제 = "지금 풀어야 하는 가장 중요한 문제" (결과 지표)
  선행문제 = 후행문제를 만들어내는 하위 원인들 (원인 지표)

• 기회-인과 그래프 예시:
  [AI 추천 개선] → [위시리스트 관련성↑] → [등록률↑] → [리텐션↑]
  각 화살표가 "인과"인지 "상관"인지 구분이 핵심

• 상관 ≠ 인과 검증법:
  RPV (Retained Probability Value): 리텐션 유저 중 해당 행동 수행 비율 → 95% 이상
  교차값: 행동 유저와 리텐션 유저의 교집합 비율 → 최대값

• DEI 반복: 실험 결과로 Description을 업데이트하는 것이 핵심
  → 한 번에 완벽한 인과 모델을 만들려 하지 말 것
```

### 3-8. If-Then-Because 가설 구조

#### 핵심 요약

| 항목 | 내용 |
|------|------|
| 출처 | Lean Startup / 실험 설계 일반론 |
| 한 문장 | 모든 제품 아이디어를 "만약(If) ___하면, ___할 것이다(Then). 왜냐하면(Because) ___이기 때문이다"라는 검증 가능한 가설로 전환하는 사고법 |
| 구조 | **If** [우리가 하는 행동], **Then** [측정 가능한 결과가 발생할 것이다], **Because** [이것이 작동하는 이유/메커니즘] |
| 언제 강력한가 | 아이디어가 남발되는 상황에서 "이거 진짜 되는 거야?"를 빠르게 판별할 때 |
| 한계 | Because가 약하면 가설 전체가 흔들림. 인과 분석(5-Whys, TDCC)과 병행 필요 |

#### 실무 적용 포인트

```
• 좋은 가설의 3요소:
  ① If: 구체적이고 실행 가능한 행동
  ② Then: 측정 가능한 지표와 목표값
  ③ Because: 이 행동이 결과로 이어지는 논리적 메커니즘

• 나쁜 가설 vs 좋은 가설:
  나쁜: "AI 추천을 넣으면 리텐션이 오를 것이다" (Because 없음, 목표값 없음)
  좋은: "If 위시리스트 기반 AI 추천을 Day 1에 노출하면,
         Then 7일 내 2회 구매 달성률이 8%에서 15%로 오를 것이다,
         Because 위시리스트 등록자의 리텐션이 34%로 비등록자(6%)의 5.7배이고,
         AI가 위시리스트 등록 행동을 앞당길 수 있기 때문이다."

• Because가 가장 중요:
  → Because가 약하면 아무리 그럴듯해도 '기도'에 불과
  → Because를 검증하는 것이 실험의 진짜 목적

• 반증 조건 필수:
  "___가 나오면 이 가설은 틀렸다" (실패 기준)
  "___가 나오면 판단 보류" (애매 기준)
```

#### 프롬프트 템플릿

```
당신은 If-Then-Because 가설 설계 전문가입니다.
아래 상황에서 검증 가능한 가설을 설계해주세요.

## 현재 상황
[챗봇 상황 카드 내용 붙여넣기]

## 분석 요청

1. **핵심 가설 2개 작성**
   각각 아래 형식으로:
   
   If: [구체적이고 실행 가능한 행동]
   Then: [측정 가능한 지표]가 [현재값]에서 [목표값]으로 변할 것이다
   Because: [이것이 작동하는 이유/메커니즘]

2. **Because 강도 평가**
   각 가설의 Because가 얼마나 근거가 있는지 평가해주세요.
   - 강함: 이미 데이터로 뒷받침되는 메커니즘
   - 보통: 논리적이지만 아직 검증 안 됨
   - 약함: 희망적 추측에 가까움

3. **반증 조건 설계**
   각 가설에 대해:
   - 실패 기준: "___가 나오면 이 가설은 틀렸다"
   - 애매 기준: "___가 나오면 판단 보류하고 ___를 추가 확인"

4. **우선순위 판단**
   두 가설 중 먼저 검증할 것을 선택. 기준:
   (1) Because의 강도 (약한 것을 먼저 검증 = 빠른 실패)
   (2) 틀렸을 때 임팩트 (큰 것을 먼저 검증 = 리스크 관리)

5. **팀에 제안할 행동 1가지**
   If-Then-Because 관점에서 이 턴에 팀이 해야 할 가장 중요한 행동과 이유.

한국어로, 팀 토론에서 3분 안에 공유할 수 있는 분량으로 작성해주세요.

```
• 토스 아하 모멘트가 바로 XYZ 가설의 구조:

• 나쁜 목표 vs 좋은 XYZ 가설:
  나쁜: "리텐션을 올린다"
  좋은: "신규 가입자의 20%가 7일 내 위시리스트를 3개 이상 등록할 것이다"

• X(%)를 정하는 방법:
  ① 현재 base rate에서 상대적 개선으로 설정
     (현재 8%라면, 2배인 16%가 목표)
  ② 업계 벤치마크 참조
  ③ CC 역산: 원하는 CC에 도달하려면 X%가 얼마여야 하는지 계산

• 가설 실패 시 배움 설계:
  "X%가 안 나왔다"로 끝내지 말고,
  "어떤 세그먼트에서는 나왔고, 어떤 세그먼트에서는 안 나왔는가?"를 분석

## 4. 프레임워크 × 미션 호환성 매트릭스

어떤 프레임워크가 어떤 미션에서 특히 강력한지 참고용으로 정리합니다.

> **이 매트릭스는 참가자에게 배포하지 않습니다.** 운영자 및 챗봇 시스템 프롬프트용입니다.

| 미션 | 🔍 고객 발견형 | 📐 구조 분해형 | 🔗 원인 추적형 | 🧪 가설 실험형 |
|------|---------------|---------------|---------------|---------------|
| M1 현황 파악 | ★★★ 디자인씽킹 (공감 인터뷰) | ★★★ MECE (이슈 분해) | ★★☆ 5-Whys (초기 탐색) | ★☆☆ |
| M2 문제 정의 | ★★★ JTBD (Job 정의) | ★★★ S-C-Q-A (문제 프레이밍) | ★★★ 5-Whys + TDCC (근본 원인) | ★☆☆ |
| M3 AI 가설 | ★★☆ JTBD (어떤 Job에 AI?) | ★★☆ CIRCLES (체계적 솔루션) | ★★☆ TDCC (인과 경로) | ★★★ If-Then-Because |
| M4 인과 설계 | ★☆☆ | ★★☆ MECE (변수 분해) | ★★★ TDCC (기회-인과 그래프) | ★★☆ XYZ (정량화) |
| M5 실험 설계 | ★★☆ 디자인씽킹 (프로토타입) | ★☆☆ | ★★☆ 5-Whys (실패 원인 사전탐색) | ★★★ XYZ + If-Then-Because |
| M6 피칭 | ★★☆ JTBD (고객 스토리) | ★★★ S-C-Q-A (피칭 구조) | ★★☆ TDCC (인과 증명) | ★★☆ XYZ (정량 목표) |

> ★★★ = 이 미션에서 가장 자연스러운 프레임워크
> ★★☆ = 보완적으로 유용
> ★☆☆ = 직접 관련성은 낮지만 사용 가능

### 매트릭스 활용법 (운영자용)

- **챗봇 사후 라벨링 시**: 팀의 행동이 어떤 유형에 해당하는지 판단할 때 참고
- **팀이 막혀있을 때**: ★★★ 프레임워크를 넌지시 힌트로 제공 가능
- **클로징 리뷰 시**: 팀이 주로 어떤 유형의 접근을 했는지 패턴 분석

## 5. 개인 LLM 활용 가이드 (참가자 공통 배포)

```
📌 개인 LLM 활용 가이드

■ 언제 사용하나요?
  매 턴, 팀 토론 전 3분간 개인 분석 시간에 사용합니다.
  
■ 어떻게 사용하나요?
  1. 챗봇의 상황 카드 내용을 복사합니다
  2. 배포받은 프롬프트 템플릿의 [현재 상황]에 붙여넣습니다
  3. 본인의 LLM(ChatGPT, Claude 등)에 입력합니다
  4. 결과 중 "팀에 제안할 행동" 항목을 팀 토론에서 공유합니다

■ 프레임워크가 2개 이상이라면?
  - Mission 1-3: 1순위(Primary) 프레임워크 프롬프트 사용
  - Mission 4-6: 2순위(Complementary) 프레임워크 프롬프트 사용
  - 물론 상황에 따라 자유롭게 바꿔도 됩니다

■ 팀 토론에서는?
  - 각자 다른 프레임워크로 같은 상황을 분석한 결과를 공유
  - "내 관점에서는 ___인데, 네 관점에서는 왜 ___라고 보는 거야?"
  - 관점이 충돌할 때가 가장 좋은 학습 기회입니다
  - 최종적으로 팀 액션 카드에는 하나의 합의된 행동을 적습니다

■ 주의사항
  - LLM의 분석을 정답으로 받아들이지 마세요. 관점을 얻는 도구입니다.
  - 턴당 3분을 넘기지 마세요. 팀 토론이 더 중요합니다.
  - 이전 턴의 결과가 누적되니, 
    상황 카드뿐 아니라 이전 결과 카드도 함께 붙여넣으면 더 좋습니다.
```
